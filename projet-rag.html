PROJET PERSONNEL — IA GÉNÉRATIVE · V7

Assistant Documentaire RAG Local V7

Un système avancé de question-réponse sur corpus documentaire privé (PDF et Word), entièrement hébergé en local sur GPU pour garantir une stricte confidentialité. La V7 introduit une architecture modulaire, un Contextual Retrieval asynchrone (asyncio + batching Ollama parallèle), un hot-swap de configuration sans redémarrage, et un chunking sémantique spécialement calibré pour les documents financiers français. La base vectorielle s'auto-alimente en continu via Watchdog : chaque nouveau document déposé sur le NAS est automatiquement ingéré, indexé et suppressible. Totalement indépendant d'APIs cloud.

Python asyncio LangChain ChromaDB Ollama (Mistral) HuggingFace BGE-M3 Gradio Contextual Retrieval & HyDE CrossEncoder Reranker Hot Swap Config CUDA GPU Watchdog

🎯 Contexte & Objectif

J'ai conçu un assistant documentaire intelligent capable de répondre à des questions en langage naturel sur un corpus de documents personnels (bulletins de paie, avis d'imposition, baux, quittances, contrats d'assurance…). Le défi : rendre des centaines de PDF et fichiers Word « interrogeables » sans hallucinations, en s'appuyant uniquement sur les passages réellement présents dans les documents — le tout en 100% local, sans aucune API cloud, pour garantir la confidentialité totale des données.

🖥️ Infrastructure Complète

📁
Source documentaire — NAS Synology

Les documents sont stockés sur un NAS Synology en réseau local (SMB). Le système accède au partage réseau \\IP_NAS_LOCAL\homes\...\Graph_Project pour scanner les dossiers et sous-dossiers automatiquement.

⚡
GPU CUDA — Accélération matérielle

Les embeddings BAAI/bge-m3 et le reranker BAAI/bge-reranker-v2-m3 tournent sur GPU CUDA pour des temps de réponse optimaux. Le LLM Mistral s'exécute lui aussi en local via Ollama.

👁️
Watchdog — Indexation temps réel

Un observateur de fichiers (watchdog) surveille en permanence le dossier source. Tout nouveau document déposé ou modifié est automatiquement ingéré et indexé sans intervention manuelle.

🔒
Hashing MD5 — Indexation incrémentale

Chaque fichier est hashé (MD5). Seuls les fichiers nouveaux ou modifiés sont réindexés, évitant tout traitement superflu. L'ancienne version d'un document est automatiquement purgée de ChromaDB avant réindexation.

🏠
100% Local — Zéro cloud

Aucune donnée ne quitte le réseau local. Pas de Gemini API, pas d'OpenAI, pas de service tiers. Le LLM (Mistral via Ollama), les embeddings et le reranker tournent intégralement sur la machine locale.

🏗️ Pipeline RAG Avancé

  1. Ingestion & Contextual Retrieval Asynchrone

    Les PDF sont parsés via PyMuPDFLoader (préservation du layout spatial des tableaux), les fichiers Word via UnstructuredWordDocumentLoader. Le Contextual Retrieval (méthode Anthropic) enrichit chaque chunk d'un résumé LLM contextuel — désormais exécuté en mode asynchrone avec asyncio.gather() et batching (5 appels Ollama simultanés). Gain de vitesse : ~5× par rapport à la version synchrone précédente.

  2. Extraction de métadonnées multi-couches

    Extraction en deux passes : 1) nom de fichier (regex multi-formats : AAAA-MM, MM/AAAA, DDMMAAAA, mois en lettres) puis 2) contenu des premières pages en fallback si les données sont incomplètes. La détection du type de document utilise un scoring multi-mots-clés (seuil de score > 0) plutôt qu'une simple correspondance — bien plus robuste pour les noms de fichiers génériques. Normalisation Unicode des accents via unicodedata.

  3. Chunking sémantique financier + Vectorisation GPU

    Le découpage utilise des séparateurs sémantiques spécifiques aux documents financiers français (\nNet à payer, \nCotisations salariales, \nRevenu fiscal…) pour ne jamais couper un tableau de cotisations en plein milieu. Taille optimisée à 800 caractères / 150 overlap. Chaque chunk est vectorisé via BAAI/bge-m3 sur GPU CUDA et stocké dans ChromaDB.

  4. HyDE & Recherche Sémantique à Fallback Progressif

    La question utilisateur est transformée en « réponse hypothétique » par le LLM (HyDE) pour maximiser la similarité vectorielle. La recherche ChromaDB applique des filtres par métadonnées (année, mois, type, catégorie) avec un fallback progressif à 3 niveaux : filtre complet → relâchement du mois → relâchement de l'année → recherche globale.

  5. Cross-Encoder Reranking Neural

    Les 20+ candidats initiaux sont réévalués en profondeur par le modèle BAAI/bge-reranker-v2-m3 (CrossEncoder multilingue sur GPU). Les 5 extraits les mieux scorés sont injectés dans le prompt final.

  6. Génération Anti-Hallucination (Mistral Local)

    Le LLM Mistral (Ollama, température 0) génère la réponse avec un prompt engineering strict : directives par type de document (salaires → « NET A PAYER EN EUROS » uniquement, impôts → « Revenu Fiscal de Référence »), règles de validation anti-hallucination, et citation systématique des sources.

  7. Mémoire conversationnelle

    L'interface conserve les 3 derniers échanges dans le contexte du prompt, permettant des questions de suivi naturelles (« Et pour le mois suivant ? », « Compare avec 2023 »).

🧠 Techniques IA Avancées

HyDE (Hypothetical Document Embeddings)

Avant la recherche vectorielle, la question utilisateur est transformée par le LLM en une « réponse hypothétique » — un extrait de document fictif qui ressemblerait à la réponse idéale. L'embedding de ce document hypothétique est beaucoup plus proche des vrais résultats pertinents dans l'espace vectoriel que la question brute.

Contextual Retrieval (inspiré Anthropic)

Pour les documents financiers critiques (bulletins de paie, avis d'imposition), chaque chunk est enrichi d'un mini-résumé contextuel généré par le LLM avant l'ingestion. Cela combat la perte de contexte inhérente au découpage, permettant au moteur de recherche de comprendre « ce chunk parle du salaire de mars 2024 » même si le chunk brut ne contient que des chiffres.

Prompt Engineering Anti-Hallucination

Le prompt système contient des directives métier par type de document : pour les salaires, seul le « NET A PAYER EN EUROS » (post prélèvement à la source) est retourné, jamais le net avant impôt ni le coût employeur. Si l'information exacte n'est pas explicitement présente dans le contexte, le système refuse de répondre plutôt que de deviner.

⚙️ Architecture Modulaire V7

La V7 refactorise le monolithe original en 5 modules indépendants, chacun avec une responsabilité unique :

hot_config.py
Hot Swap de Configuration

Thread daemon surveillant config.json toutes les 30s. Modifier le modèle Ollama, la taille des chunks ou le prompt sans redémarrer le pipeline ni perdre l'état Watchdog.

metadata_extractor.py
Extraction Multi-couches

Parsing du nom de fichier en priorité, fallback automatique sur le contenu des premières pages. Scoring multi-mots-clés pour la détection du type de document.

context_enricher.py
Contextual Retrieval Async

Batching asynchrone via asyncio.gather(). 5 appels Ollama simultanés par batch (~5× plus rapide). Fallback automatique sur préfixe statique pour les documents non prioritaires.

chunk_strategy.py
Chunking Sémantique

Séparateurs financiers prioritaires (\nNet à payer, \nTotal cotisations…) pour préserver la cohérence des lignes de tableau. Filtrage des micro-chunks (< 40 caractères).

💡 Apprentissages Clés

  • ✓ 800 caractères / 150 overlap est un meilleur réglage que 600/200 pour les documents financiers : les tableaux de cotisations tènent dans un seul chunk et l'overlap excessif (33%) ne polluait que les résultats.
  • ✓ PyMuPDFLoader préserve le layout spatial des tableaux bien mieux que UnstructuredPDFLoader, qui casse les montants en éléments isolés incompréhensibles.
  • ✓ Le reranking CrossEncoder après la recherche vectorielle améliore drastiquement la pertinence — les embeddings seuls ne suffisent pas pour des requêtes financières précises.
  • ✓ Le Contextual Retrieval LLM synchrone bloquait ~40s pour 20 chunks. L'implémentation asyncio + batching (5 appels simultanés) ramène ce délai à ~8-10s sans changer la qualité du contexte généré.
  • ✓ Le fallback progressif sur les filtres de métadonnées est indispensable : un filtre trop strict retourne zéro résultat, un filtre absent noie la réponse dans du bruit.
  • ✓ L'extraction de métadonnées depuis le seul nom de fichier échoue pour les fichiers génériques (ex: doc_0043.pdf). Le fallback sur le contenu des premières pages réduit les cas type=other de ~40%.
  • ✓ Le hot-swap de configuration via un thread de surveillance de config.json évite les redémarrages Watchdog — critique en production continue sur un NAS.
  • ✓ Afficher les sources dans l'UI est essentiel pour établir la confiance utilisateur et permettre la vérification humaine.
