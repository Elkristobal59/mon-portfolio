Tesla Inventory Monitor :
Le Bot qui ne Rate Aucune Opportunité

Je cherche à acheter une Tesla. Le problème ? Les véhicules en stock disparaissent en quelques heures, souvent avant même que je ne rafraîchisse la page. Ma solution : un bot de monitoring autonome qui surveille l'inventaire Tesla France 24h/24 depuis mon serveur Proxmox, contourne les protections anti-bot d'Akamai, et m'envoie une notification Telegram instantanée dès qu'un nouveau véhicule apparaît.

Python 3.11 nodriver (CDP) Playwright curl_cffi Docker Compose Xvfb Telegram Bot API Proxmox

⚡ Le Défi : Pourquoi c'est Difficile

Surveiller l'inventaire Tesla ne se résume pas à un simple requests.get(). Le site est protégé par Akamai Bot Manager, l'un des systèmes anti-bot les plus sophistiqués au monde. Chaque requête est analysée en temps réel : empreinte TLS, fingerprint JavaScript, cohérence des headers, type d'IP (résidentielle vs. datacenter), et même les mouvements de souris.

🛡️

Akamai Bot Manager

Analyse TLS, JavaScript fingerprinting, détection de headless browser, challenge cookies en temps réel.

🌐

IPv6 vs IPv4

Le serveur sortait en IPv6 par défaut, une famille d'IP traitée différemment par Akamai. Invisible depuis un PC classique.

👻

Headless Detection

Même le mode --headless=new de Chrome est détecté par les capteurs Akamai dans un environnement serveur.

🏗️ Architecture Anti-Détection Multicouche

Le bot utilise une stratégie en cascade : si la méthode principale échoue, il bascule automatiquement sur la suivante. Cette résilience garantit une disponibilité maximale.

PRIORITÉ 1

nodriver — Contrôle Chrome via CDP

nodriver pilote Google Chrome directement via le Chrome DevTools Protocol, sans laisser les traces détectables de Playwright ou Selenium. Combiné à Xvfb (un écran virtuel), Chrome tourne en mode fenêtré complet sur le serveur — une empreinte identique à un utilisateur réel sur un PC Windows.

💡 Astuce clé : après avoir établi la session, le bot appelle directement l'API Tesla via un XHR synchrone depuis le contexte du navigateur. Les cookies Akamai sont automatiquement inclus.

FALLBACK 2

curl_cffi — Empreinte TLS Chrome

Appel API direct imitant parfaitement la signature TLS de Chrome 124, sans navigateur. Ultra-rapide (~200ms).

FALLBACK 3

Playwright + Stealth — Navigation complète

Navigateur automatisé avec injection stealth, interception réseau et simulation d'attente humaine.

🔧 Les Obstacles Techniques (et leurs Solutions)

Blocage IPv6 invisible

Le serveur Proxmox sortait en IPv6 (2a02:8429:...) au lieu de l'IPv4 résidentielle. Akamai bloquait silencieusement cette famille d'IP.

✅ Solution : sysctls: net.ipv6.conf.all.disable_ipv6=1 dans Docker Compose.

Headless Chrome détecté

Même le mode --headless=new de Chrome était repéré par les capteurs JavaScript d'Akamai (absence de plugins, screen dimensions, etc).

✅ Solution : Xvfb (écran virtuel 1920×1080) + Chrome en mode fenêtré complet = empreinte identique à un vrai PC.

Bug CDP nodriver (sameParty)

L'interception réseau via CDP crashait à cause d'un champ cookie supprimé dans Chrome 147 (sameParty), incompatible avec le parser de nodriver.

✅ Solution : Appel XHR direct à l'API Tesla depuis le contexte navigateur — les cookies Akamai sont déjà validés.

Boucle async incompatible

nodriver est async, mais le script principal est synchrone. asyncio.run() échouait dans certains contextes.

✅ Solution : nest_asyncio.apply() pour autoriser les boucles async imbriquées.

🐳 Déploiement Docker sur Proxmox

Le bot tourne 24h/24 dans un conteneur Docker sur mon serveur Proxmox, avec restart: unless-stopped. Il survit aux redémarrages du serveur et aux mises à jour automatiquement.

# docker-compose.yml
services:
  tesla-scraper:
    build: .
    container_name: tesla-scraper
    restart: unless-stopped
    shm_size: '1gb'  # Mémoire partagée pour Chrome
    sysctls:
      - net.ipv6.conf.all.disable_ipv6=1  # Force IPv4
    volumes:
      - ./.env:/app/.env  # Secrets Telegram
      - ./stock_vins.json:/app/stock_vins.json  # Cache persistant
# Dockerfile — Chrome + Xvfb (faux écran)
CMD xvfb-run --auto-servernum \
  --server-args="-screen 0 1920x1080x24" \
  python tesla_stock_monitor.py

"Ne jamais rater une opportunité, c'est automatiser sa veille."

Ce projet m'a confronté aux réalités du web scraping moderne : les protections anti-bot de niveau enterprise (Akamai, Cloudflare) ne se contournent pas avec de simples requêtes HTTP. Il faut comprendre le fonctionnement des empreintes TLS, du fingerprinting JavaScript, et maîtriser les subtilités réseau (IPv4/IPv6) qui passent inaperçues en développement local mais deviennent critiques en production sur un serveur.
